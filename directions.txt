High-level architecture
Framework: Spring Boot (REST), Spring Security with JWT, Spring Data JPA.
Layers:
Controller layer: Auth, SuperAdmin, Admin, Donneur (donor), Patient, Notification, Public, Chatbot.
Service layer: AuthService, DashboardService, AnalyticsService, DonneurService, DemandeSangService, StockService, GamificationService, NotificationService, CampagneService, ChatbotService, SuperAdminService.
Repository layer: JPA repositories for entities Don, Donneur, Hopital, StockSang, DemandeSang, CampagneDon, Badge, DonneurBadge, Notification, RapportMensuel, StatistiquesQuotidiennes, Utilisateur.
Security: JwtAuthenticationFilter, JwtUtil, SecurityConfig, UserDetailsServiceImpl, Custom access denied and entry point handlers.
Schedulers: ScheduledTasks.
Chatbot helpers: AIService, IntentDetector, FAQHandler, ActionHandler.
DTOs: comprehensive request/response objects for all features.
Enums: Roles, sexes, groups, statuses, urgency, notification types/priorities, etc.
Docs: API_DOCUMENTATION.md and Postman collection provided.
Domain model (inferred from package model)
Utilisateur: base user, fields include id, email, motDePasse, role (enum Role), pointsTotal, may be linked to Hopital for ADMIN. Used by all auth and identity flows.
Donneur: donor profile tied to a Utilisateur. Fields include cin, dateNaissance, sexe, groupeSanguin (GroupeSanguin), poids, adresse, ville, antecedentsMedicaux, eligible (Boolean), and historical donations.
Hopital: hospital unit with identity fields (nom, adresse, ville, region, telephone, email), capaciteStockage, description, statut (StatutHopital). Linked to Admins and StockSang.
StockSang: per-hospital stock quantities per group, with NiveauStock (critical/normal/high) handled in service.
Don: donation records (donneur, hopital, quantiteMl, notes, dates for don/validation/peremption, numeroPoche, statut StatutDon).
DemandeSang: patient demand (groupeSanguinDemande, quantiteDemandee, urgence, statut StatutDemande, patientId or Utilisateur as patient, dateDemande, hopital).
CampagneDon: donation campaigns (nationale boolean or hopitalId, dates, actifs).
Badge, DonneurBadge: gamification with NiveauBadge.
Notification: user-targeted notifications with TypeNotification, PrioriteNotification, flags lue, timestamps, lienAction.
RapportMensuel, StatistiquesQuotidiennes: analytics aggregates.
Security model
JWT-based authentication:
Auth endpoints are public.
All others protected by roles with @PreAuthorize.
Role mapping:
/api/superadmin/**: SUPER_ADMIN
/api/admin/**: ADMIN or SUPER_ADMIN
/api/donneur/** and /api/patient/**: USER, ADMIN, SUPER_ADMIN
/api/notifications/**: isAuthenticated
/api/public/**: open
JwtAuthenticationFilter extracts and validates token; JwtUtil handles creation/parsing; UserDetailsServiceImpl loads users by email.
Scheduled/background tasks
ScheduledTasks likely performs:
Expiration handling for Don (PERIME after 42 days).
Stock threshold recalculation and urgent notifications.
Aggregating daily/monthly analytics.
Chatbot
ChatbotController endpoints:
POST /api/chatbot/message: processes message with optional user context (utilisateurId from auth if present) via ChatbotService.
GET /api/chatbot/suggestions: suggestions and quick actions for UI.
GET/DELETE /api/chatbot/historique: get/clear conversation history (by sessionId).
GET /api/chatbot/health: health info for chatbot service.
ChatbotService composes AIService, IntentDetector, FAQHandler, ActionHandler to return ChatMessageResponse. It can trigger app actions by intent (e.g., checking eligibility, listing campaigns, etc.).

Side effects/business rules worth noting for frontend behavior
Donation validation by admin:
On validation, stock is incremented and donor receives points and maybe badges.
Poche number generated; peremption set to now + 42 days.
Donor eligibility gate enforced at donor donation declaration.
Admin must be associated to a Hopital to access dashboard and lists.
Global and targeted notifications propagate to users; unread counts are available.
Urgent demands endpoint aggregates high urgency items (for admin alert banners).
Stocks critical endpoint for quick admin action.
SuperAdmin must validate new hospitals before they can be used.
Campaigns can be national (superadmin) or local (admin); donors can participate.
