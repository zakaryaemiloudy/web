## 1. Project Overview

**BKS (Blood Bank System)** is a REST API backend built with **Spring Boot 3.3** for managing blood donations in Morocco. It serves three types of users:

- **Super Admins** — Platform-level administrators who manage hospitals and global campaigns
- **Hospital Admins (ADMIN)** — Hospital-level managers who validate donations, process requests, and manage their hospital's stock
- **Regular Users (USER)** — Donors and patients who register donations and request blood

Core features:
- JWT-based authentication and role-based access control
- Blood donation tracking with eligibility verification
- Blood stock management per hospital and per blood group
- Blood demand/request processing with urgency levels
- Gamification: points and 8-tier badge system for donors
- Notification system (user-level, role-level, hospital-level, global)
- Donation campaigns (hospital-scoped and national)
- AI-powered chatbot (Groq/LLaMA) with FAQ and intent detection
- Automated background jobs (expired donations, eligibility updates, stats)

**Base URL:** `http://localhost:8080`

All API routes start with `/api/`.

## 4. Authentication & Security

### How Authentication Works

1. User calls `POST /api/auth/connexion` with email + password
2. Server responds with a **JWT token** (Bearer type)
3. All protected endpoints require the header:
   ```
   Authorization: Bearer <token>
   ```
4. Token expires after **24 hours**

### Token Claims

The JWT contains:
- `sub` — user email (used as username)
- `userId` — user's database ID
- `role` — user role string (e.g., `USER`, `ADMIN`, `SUPER_ADMIN`)
- `exp` — expiration timestamp

### Security Rules Summary

| Route Pattern | Access |
|---|---|
| `OPTIONS /**` | Public (CORS preflight) |
| `/api/auth/**` | Public |
| `/api/public/**` | Public |
| `/api/superadmin/**` | SUPER_ADMIN only |
| `/api/admin/**` | ADMIN or SUPER_ADMIN |
| `/api/donneur/**` | USER, ADMIN, or SUPER_ADMIN |
| `/api/patient/**` | USER, ADMIN, or SUPER_ADMIN |
| `/api/notifications/**` | Any authenticated user |
| `/api/chatbot/**` | Public (but user ID extracted if logged in) |
| All others | Any authenticated user |

### Error Responses for Auth Failures

**401 Unauthorized** (missing or invalid token):
```json
{
  "timestamp": "2024-01-15T10:00:00",
  "status": 401,
  "error": "Non autorisé",
  "message": "Token JWT manquant ou invalide. Veuillez vous connecter.",
  "path": "/api/endpoint"
}
```

**403 Forbidden** (insufficient role):
```json
{
  "timestamp": "2024-01-15T10:00:00",
  "status": 403,
  "error": "Accès refusé",
  "message": "Vous n'avez pas les permissions nécessaires pour cette action. Vérifiez votre rôle.",
  "path": "/api/endpoint"
}
```

## 5. Roles & Permissions

There are exactly **3 roles** in the system:

| Role | String value | Description |
|---|---|---|
| `SUPER_ADMIN` | `"SUPER_ADMIN"` | Platform-wide administrator |
| `ADMIN` | `"ADMIN"` | Hospital administrator (must be linked to a hospital) |
| `USER` | `"USER"` | Regular user (donor and/or patient) |

**Important notes:**
- A `SUPER_ADMIN` can access all ADMIN and USER routes too
- An `ADMIN` must have a linked hospital (`hopitalId` not null) — several admin endpoints use `auth.getHopital().getId()` automatically
- A `USER` can be both a donor (has Donneur profile) and a patient (can submit blood requests)
- Spring Security stores roles as `ROLE_USER`, `ROLE_ADMIN`, `ROLE_SUPER_ADMIN` internally, but the JWT and database store just `USER`, `ADMIN`, `SUPER_ADMIN`

## 8. DTOs — Request & Response Shapes

### ConnexionRequest
```json
{
  "email": "user@example.com",
  "motDePasse": "password123"
}
```
Validations: `email` must be valid email format (not blank), `motDePasse` not blank.

### InscriptionRequest
```json
{
  "email": "user@example.com",
  "motDePasse": "password123",
  "nom": "Alami",
  "prenom": "Karim",
  "telephone": "0612345678",
  "role": "USER",
  "hopitalId": null
}
```
Validations:
- `email`: valid email, not blank
- `motDePasse`: min 6 characters, not blank
- `nom`, `prenom`: not blank
- `telephone`: exactly 10 digits (`^[0-9]{10}$`)
- `role`: not null, must be `USER`, `ADMIN`, or `SUPER_ADMIN`
- `hopitalId`: optional, required only for ADMIN role

  "token": "eyJhbGci...",

#### `POST /api/auth/inscription`

**Required role:** `USER`, `ADMIN`, or `SUPER_ADMIN`

#### `POST /api/donneur/profil`

Create a donor profile for the currently authenticated user. Must be called after registration to enable blood donation.

**Headers:** `Content-Type: application/json`, `Authorization: Bearer <token>`

**Request Body:** Donneur entity fields (the `utilisateur` link is set automatically from JWT)

**Error Responses:**

**Eligibility rules checked at creation:**
- Age: 18–65 years (based on `dateNaissance`)
- Weight: at least 50 kg
- Last donation: at least 90 days ago (irrelevant at creation unless donor was previously registered)

#### `PUT /api/admin/dons/{id}/valider`

Validate a pending donation.

**Headers:** `Authorization: Bearer <token>`

**Path Variable:** `id` — donation ID (Long)

**Success Response:** `200 OK` — `Don` entity (updated)

**What happens on validation:**
1. `statut` set to `VALIDE`
2. `valideParId` set to admin's user ID
3. `dateValidation` set to now
4. `datePeremption` set to now + 42 days
5. `numeroPoche` generated as `"POCHE-{System.currentTimeMillis()}"`
6. Blood stock incremented by `quantiteMl` for the donor's blood group at the hospital
7. 10 points awarded to the donor
8. Badge check triggered — donor may earn new badges

**Error Responses:**
- `400 Bad Request` — Donation not found

---

#### `PUT /api/admin/dons/{id}/rejeter`

Reject a pending donation.

**Headers:** `Content-Type: application/json`, `Authorization: Bearer <token>`

**Path Variable:** `id` — donation ID (Long)

**Request Body:** Plain string (rejection reason)
```
"Test results indicate incompatibility"
```
Send as raw string in request body, not as JSON object. Content-Type should be `application/json` and body should be a JSON string: `"reason text"`.

**Success Response:** `200 OK` — `Don` entity (updated with `statut = REJETE` and `notes = raison`)

#### `GET /api/admin/demandes`

**Headers:** `Authorization: Bearer <token>`

**Success Response:** `200 OK` — Array of `DemandeSangResponse`

#### `PUT /api/admin/demandes/{id}/traiter`

**Headers:** `Authorization: Bearer <token>`

**Path Variable:** `id` — request ID (Long)

**Query Parameter:** `statut` — new status value (StatutDemande enum string)

Example:

## 12. Gamification System

### How Points Work
- Each validated donation awards **10 points** to the donor
- Points are stored in `Utilisateur.pointsTotal`
- Points accumulate — they are never deducted

### How Badges Work
- After each donation validation, the system checks if the donor qualifies for new badges
- Badges are based on `Donneur.nombreDonsTotal`
- Each badge tier is awarded once — duplicates are prevented

### Badge Tiers

| Premier Don | BRONZE | 1 |
| Donneur Régulier | ARGENT | 5 |

## 15. Frontend Integration Checklist

Use this checklist to ensure complete and correct integration:

### Auth Flow
- [ ] On login, store `token`, `role`, `id`, `hopitalId` from `AuthResponse`
- [ ] Attach `Authorization: Bearer <token>` to every protected request
- [ ] Handle 401 responses globally — redirect to login and clear stored token
- [ ] Handle 403 responses — show "Access denied" message
- [ ] Use `GET /api/auth/valider-token` on app startup to verify stored token is still valid

### Role-Based UI
- [ ] Show/hide UI sections based on `role` from stored auth data
- [ ] `SUPER_ADMIN`: full access to hospital management, global stats, national campaigns
- [ ] `ADMIN`: hospital dashboard, donation validation, request processing, local campaigns
- [ ] `USER`: donor profile, donation submission, blood requests, notifications

### Hospital Selection
- [ ] Fetch hospitals from `GET /api/public/hopitaux` for registration dropdowns — no auth needed
- [ ] Only hospitals with `statut = VALIDE` are returned

### Donor Registration Flow
1. User registers via `POST /api/auth/inscription` with `role: "USER"`
2. User logs in via `POST /api/auth/connexion`
3. User creates donor profile via `POST /api/donneur/profil`
4. User can now submit donations

### Admin Workflow
1. SuperAdmin creates hospital: `POST /api/superadmin/hopitaux`
2. SuperAdmin validates hospital: `PUT /api/superadmin/hopitaux/{id}/valider`
3. Register admin user with `role: "ADMIN"` and `hopitalId` pointing to validated hospital
4. Admin logs in — all admin endpoints auto-scope to their hospital

### Date/Time Format
- All `LocalDateTime` fields use ISO 8601 format: `"2024-01-15T10:30:00"` (no timezone suffix)
- All `LocalDate` fields use: `"1990-05-15"`
- The backend operates in UTC timezone

### Enum Values
- Always send enum values as UPPERCASE strings exactly as documented
- Example: `"groupeSanguin": "A_POSITIF"` — not `"a_positif"` or `"A+"`

### Pagination
- **No pagination is implemented** — all list endpoints return full arrays
- Implement client-side pagination/filtering where needed

### Stock Display
- For each blood group in `StockResponse`, use `niveauStock` to color-code:
  - `CRITIQUE` → red
  - `ALERTE` → orange
  - `NORMAL` → yellow
  - `OPTIMAL` → green
- `pourcentageDisponible` = `(quantiteDisponible * 100.0) / (seuilAlerte * 2)`

### Notifications
- Poll `GET /api/notifications/count` periodically (e.g., every 30s) to show unread badge
- On notification click, call `PUT /api/notifications/{id}/lire` to mark it read
- Use `type` and `priorite` fields to style notification items

### Campaign Participation
- Check `statut` before allowing participation — only `EN_COURS` campaigns are joinable
- `progressionPourcentage` = `(nombreParticipants * 100.0) / objectifDonneurs`

### Chatbot Integration
- Generate a UUID as `sessionId` when the chat window opens
- Pass the same `sessionId` in every message for conversation context
- Use `GET /api/chatbot/suggestions` to pre-populate quick action buttons
- The chatbot endpoint is public — works for both logged-in and anonymous users

### Error Display
- Parse the `details` object in validation errors (400) and show per-field messages
- Parse the `message` field in runtime errors (400) for user-friendly display
- All error responses include `timestamp`, `status`, `error`, and either `message` or `details`

### Content-Type
- Always set `Content-Type: application/json` for POST and PUT requests with a body
- For `PUT /api/admin/dons/{id}/rejeter`, the body is a JSON string (e.g., `"reason text"`)

### CORS
- The backend allows credentials (`withCredentials: true` for axios or equivalent)
- Allowed origins: `http://localhost:4200`, `http://localhost:3000`, `http://localhost:8080`
